# 写在前面

“什么情况下需要 commit”看起来像一个很基础的问题，但真正做团队协作时，这件事直接决定后面的 review、回滚、排错和历史可读性。

很多仓库历史之所以难看，并不是因为大家不会写代码，而是因为 commit 的边界感太差。

## commit 的本质

我更愿意把 commit 理解成一句话：

**把一个独立的、可解释的变更单元保存下来。**

这里有三个关键词：

- 独立
- 可解释
- 可回滚

如果一组改动满足不了这三点，通常就还不适合提交。

## 什么时候应该 commit

### 1. 一个明确目标已经闭环了

比如：

- 修掉一个具体 bug
- 新增一个完整但较小的功能点
- 完成一次独立的重构
- 补齐一组相关测试

这时候 commit 的边界通常比较清晰，因为它回答的是“这次到底完成了什么”。

### 2. 代码已经处于可运行状态

不是每次都必须完美，但至少要满足一个底线：

- 不明显破坏现有流程
- 关键测试能过，或者至少自己知道为什么没跑
- 同事看到这笔提交，不会完全接不下去

如果 commit 下去的状态本身就是半残的，那历史记录很快就会变成噪音。

### 3. 你希望未来能回到这个点

这是个很实用的判断方式。

如果你心里已经出现了“这一阶段先存一下比较安全”，那往往说明它已经形成了一个足够独立的节点。

## 什么时候不该急着 commit

### 1. 改动目标还混在一起

比如同一把改动里同时出现：

- 修 bug
- 改命名
- 改样式
- 顺手升级依赖

这种提交最难 review，也最难回滚。

正确做法通常不是“先提交再说”，而是先拆开。

### 2. 只是临时试验

如果你只是为了验证一个方向，代码里还有明显的调试痕迹、注释残留、无意义日志，那更适合先在本地继续整理，而不是急着留下历史。

### 3. 你自己都说不清这笔改动干了什么

这是个很好的自检标准。

如果你没法用一句清楚的话描述这次提交，那很可能说明它的边界还没收好。

## 我常用的判断标准

我通常会在 commit 前问自己这几个问题：

### 这笔提交能不能用一句话说清楚？

比如：

- `fix: 修复筛选条件重置后分页没有恢复的问题`
- `feat: 为文章页增加最新文章模块`
- `refactor: 拆分请求层错误处理逻辑`

如果一句话说不清，多半还应该继续整理。

### 这笔提交能不能单独 review？

如果 review 的人必须在脑子里同时处理三四类问题，那这笔提交大概率太大了。

### 这笔提交如果回滚，会不会很痛？

如果回滚会顺带带走很多本不该一起回滚的东西，那说明提交边界不合理。

## commit 太大和太碎，都有问题

很多人只知道“不要太大”，但其实“太碎”也会让历史变差。

像这种：

- `fix typo`
- `update`
- `again`
- `final fix`

虽然每笔都很小，但没有信息量，最后一样不可读。

所以关键不是“大或小”，而是**是否构成一个完整且有意义的变更单元**。

## 对个人项目和团队项目的差别

个人项目里，你可以对中间态稍微宽松一点，因为历史主要是给未来的自己看。

但只要涉及团队协作，commit 的要求就应该明显更高，因为它不仅影响你自己，还影响：

- code review
- blame
- cherry-pick
- 回滚
- 发布排障

## 一个简单可执行的结论

如果你不确定该不该 commit，可以先用这个标准：

**当一组改动已经完成一个明确目标，并且你愿意让别人看到这一状态时，就可以 commit。**

反过来，如果它还是混乱的、临时的、说不清目标的，那就先别急着提交。

## 结论

commit 最重要的不是频率，而是边界。

如果提交里顺手夹带了状态管理修补或工程迁移，边界通常就会变差。像 [[vue-reset-ref-state]] 和 [[vue-cli-to-vite-migration]] 这种主题，更适合拆成独立提交处理。

好的 commit 不是“我改了一堆东西”，而是“我完成了一件清楚的事”。

一旦用这个标准去看，你会发现很多本来想一把提交的改动，其实都应该再拆一层；而很多迟迟不敢提交的阶段性成果，其实已经可以安全落库了。
